# Structured-block markdown renderer — byte-level spec

**Applies to:** `ch.v0.1` structured-block.
**Implementations:** Python (`packages/interchange-spec/python/contexthub_interchange/renderer.py`) and TypeScript (`packages/interchange-spec/ts/src/renderer.ts`).
**Contract:** Both implementations MUST produce byte-identical output for every valid `StructuredBlockV0` input. This is verified by `tests/test_golden.py` and `tests/golden.test.ts` against fixtures in `packages/interchange-spec/fixtures/structured-blocks/`.

If you change this spec, change both renderers in the same PR, regenerate all `*.expected.md` fixtures, and bump `ch.v0.1` → `ch.v0.2`.

---

## Algorithm (deterministic)

Input: validated `StructuredBlockV0`. Output: `string`.

1. Build an empty list `sections`.
2. For each of the five sections in order below, if its source list is non-empty, append one rendered section string to `sections`.
3. If `sections` is empty, return the empty string `""`. Do not emit a lone newline.
4. Otherwise, join `sections` with a single `"\n"` (a blank line between sections).
5. Strip trailing newlines, then append exactly one `"\n"`.
6. Apply Unicode NFC normalization to the result.
7. Return the result.

### Section rendering rules

All section bodies end with `"\n"` (so that joining with `"\n"` produces a blank line between sections).

**Decisions.** Header line: `"## Decisions\n\n"`. Then for each decision in order:
```
- **{title}** \u2014 {rationale}\n
```
(The separator is U+2014 EM DASH, bounded by ASCII spaces.)

**Artifacts.** Header: `"## Artifacts\n\n"`. Artifacts are separated by one blank line. Each artifact:
```
### {name} ({kind})\n
```{language-or-empty-string}\n
{body}\n        <- appended if body does not already end with "\n"
```\n
```
`language` comes from `artifact.language`; when missing or empty, the opening fence is a bare triple backtick followed by newline.

**Open Questions.** Header: `"## Open Questions\n\n"`. For each question:
```
- {question}\n
  _Context: {context}_\n    <- only if `context` is present and non-empty
```

**Assumptions.** Header: `"## Assumptions\n\n"`. For each assumption string:
```
- {assumption}\n
```

**Constraints.** Header: `"## Constraints\n\n"`. For each constraint string:
```
- {constraint}\n
```

### Invariants

- No escaping. Markdown metacharacters inside user-supplied strings are emitted verbatim. The renderer is a pure mechanical transform; it is the summarizer's job to produce text that renders safely.
- No HTML output. Always Markdown.
- Field order inside each structured item is fixed by the schema, not by JSON input order.
- Trailing whitespace inside user strings is preserved (authored intent).
- NFC normalization is applied once, at the very end, so that fixture input that uses decomposed sequences produces composed output. Do not pre-normalize individual fields.
- Empty output for an all-empty structured block is the empty string (zero bytes), not a single newline.

## Golden-fixture test protocol

1. For every fixture in `fixtures/structured-blocks/<name>.json`, read and validate against the schema.
2. Render with the Py implementation; read `<name>.expected.md` bytes; assert equal.
3. Render with the TS implementation; assert equal to the same expected bytes.
4. If either renderer disagrees with the expected bytes, CI fails.
5. When adding a new fixture, you MUST commit both the JSON and the `.expected.md`. The `.expected.md` is generated by running `uv run ch-render-golden --write` or `pnpm --filter @contexthub/interchange-spec run golden:write`.

Drift between Py and TS is the single class of bug this test is designed to catch. Treat a golden-fixture failure as "stop everything, diagnose" — not "regenerate and commit."
